## 1. Resumen Ejecutivo

La bitacora anterior (`docs/bitacora_2026-04-21.md`) queda como historico de la sesion v26-v29. Al 24 de abril de 2026, el sistema evoluciono hasta el commit local `v37`, mientras que el `index.html` publico apunta al deployment Apps Script `@49` con cache-buster `v=43`.

El proyecto esta operativo, pero hay una brecha importante entre:

- el ultimo commit documentado en Git (`v37`),
- el deploy activo referenciado por GitHub Pages (`@49`),
- y los cambios locales sin commit en `Admin.gs`, `Client.html`, `Survey.gs`, `AppIndex.html` e `index.html`.

Esto hace que la prioridad sea cerrar la trazabilidad version/deploy antes de seguir agregando funcionalidad.

### Cambios arquitectonicos vigentes

- El dashboard ya no depende solo de `BASE_ANALITICA` en vivo: combina `SnapshotData.html` para historico 2024+2025 con filas live desde `BASE_ANALITICA`.
- Se usa cache chunked para superar el limite de 100 KB de `CacheService`.
- La version de cache se invalida con `DASH_CACHE_VERSION` en `PropertiesService`.
- El cliente precarga filas compactas con `getAnalyticRowsForDashboard()` y recalcula filtros localmente cuando puede.
- `submitSurvey()` ya no ejecuta un rebuild completo; agrega incrementalmente a `BASE_ANALITICA` y `RESPUESTAS_LONG` con `appendResponsesToAnalytics_()`.

## 6. Desafios Pendientes

### Criticos

| ID | Descripcion | Impacto | Accion recomendada |
|---|---|---|---|
| P-01 | `rebuildAnalytics()` no recibe token ni valida rol. | Alto: cualquier llamada expuesta por `google.script.run` podria ejecutar un rebuild si el frontend la invoca. | Cambiar firma a `rebuildAnalytics(sessionToken)`, exigir `requireRole_(sessionToken, ['admin'])` y actualizar llamadas del cliente. |
| P-02 | `rebuildAnalytics()` depende de `getSheet_(APP_CFG.SHEETS.LONG)`. | Alto: si falta `RESPUESTAS_LONG`, falla con `No existe la hoja`. | Usar `syncHeaders_(APP_CFG.SHEETS.LONG, longHeaders)` antes de escribir. |
| P-03 | Brecha version/deploy: Git v37, iframe `v=43`, deploy Apps Script `@49`, cambios locales sin commit. | Alto: no hay trazabilidad clara entre produccion y codigo fuente. | Cerrar cambios locales, commitear, crear deployment con descripcion y documentar ID/version. |
| P-04 | Pipeline de deploy sigue manual. | Alto: ya provoco regresiones de version visible. | Crear script `deploy.ps1` o checklist automatizado que ejecute push, deploy, actualice `index.html`, verifique deploy y prepare commit. |
| P-05 | `.gitignore` no cubre todos los artefactos actuales. | Alto: riesgo de commitear ZIP/CSV/GSHEET o datos sensibles/pesados. | Agregar reglas para `*.zip`, `*.gsheet`, `data/*.csv` segun politica, `2024encuesta/` y archivos temporales de Office. |

## 7. Protocolo Recomendado de Cierre de Cambios

Antes de publicar:

```powershell
git status -sb
git diff --stat
```

Para desplegar Apps Script:

```powershell
npx clasp push -f
npx clasp deploy --description "vXX descripcion breve"
npx clasp deployments
```

Luego:

1. Copiar el nuevo deployment ID de `clasp deployments`.
2. Actualizar `index.html` con el nuevo `/exec` y el cache-buster correspondiente.
3. Verificar que la URL publica carga el nuevo deployment.
4. Ejecutar smoke test: login, dashboard, filtros, Respuestas, Cuestionario, Invitaciones.
5. Commitear todos los archivos fuente relevantes.

Comando de commit sugerido cuando el arbol este limpio de artefactos:

```powershell
git add Admin.gs Client.html Survey.gs AppIndex.html index.html docs/bitacora_2026-04-24.md
git commit -m "docs: actualizar bitacora operativa"
git push origin main
```
